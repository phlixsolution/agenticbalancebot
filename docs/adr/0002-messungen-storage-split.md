# ADR-0002: Measurement data split between NextCloud and git

## Context

Every commissioning session produces serial recordings and derived files
(processed CSVs, plots). Two collaborators work from separate machines and
both need access to the full dataset.

Storing all measurement data in git has two problems:

1. **Size.** Raw recordings grow with every session. Git is not designed
   for large or frequently changing binary/text data; the repo would bloat
   and clones would become slow.
2. **Noise.** Intermediate files (processed CSVs, plots) are derived
   artifacts. Tracking them in git creates churn with no informational gain
   beyond what the raw data already contains.

Alternatives considered:

- **Git LFS.** Adds infrastructure complexity (LFS server or paid GitHub
  quota) and still tracks every intermediate file.
- **One collaborator owns the data, shares on request.** Not symmetric;
  breaks when that collaborator is unavailable.
- **Store everything in git.** Rejected for the reasons above.
- **NextCloud for shared data, git only for curated results.** Chosen.

## Decision

```
messungen/
  raw/          Serial recordings from acquire.py    — NextCloud only
  processed/    Output of process.py                 — NextCloud only
  plots/        Output of analyze.py                 — NextCloud only
  archive/      Legacy files from old scripts        — NextCloud only
  results/      Committed summaries (key plots,      — tracked in git
                tables, final figures)
```

Both collaborators mount the NextCloud folder locally so the full path
is identical on each machine. `.gitignore` excludes `raw/`, `processed/`,
`plots/`, and `archive/`.

Only curated outputs go into `messungen/results/` and are committed —
e.g. a final deadzone identification plot, a CoG measurement table.

### File naming convention

```
<phase>_<mode>_YYYYMMDD_HHMMSS.csv
```

| Field | Values | Example |
|---|---|---|
| `phase` | `phase0` … `phase7` | `phase3` |
| `mode` | `balance`, `deadzone`, `imu`, `carrier` | `deadzone` |
| timestamp | auto-generated by `acquire.py` | `20260413_143000` |

One-second timestamp granularity is sufficient to avoid filename
collisions between simultaneous sessions.

## Consequences

**Positive**
- Repo stays lean; clones remain fast regardless of how many sessions run.
- Both collaborators always have the same dataset without manual transfers.
- Derived artifacts (processed, plots) can be regenerated from raw at any
  time; there is no value in versioning them.

**Negative**
- No git history for measurement data. Traceability relies on the filename
  timestamp and the NextCloud sync log rather than git blame.
- If NextCloud is unavailable, a collaborator cannot access raw data.
  Mitigated by local caches (NextCloud syncs to disk).

**Neutral**
- The `results/` convention means "committed" is a deliberate editorial
  act, not an accident. Each commit to `results/` should note which
  commissioning phase produced the file.
